# Consult Note

Version: 0.1
Target Platform: Windows
Application Type: Single-user Desktop App

---

# 1. Overview

Consult Note는 장기렌트 상담 업무를 위한 개인용 상담 기록 및 파일 관리 프로그램이다.

본 프로그램은 회사 CRM/DB를 대체하지 않는다.
회사의 고객 문의 DB는 기존처럼 사용하며, Consult Note는 상담원이 개인 PC에서 Word 문서와 고객별 폴더로 관리하던 상담 기록, 견적 이미지, 면허증, 계약 관련 파일을 더 빠르게 검색하고 열람하기 위한 보조 도구이다.

---

# 2. Problem

현재 업무 방식은 다음과 같다.

1. 회사 페이지에 고객 문의 DB가 들어온다.
2. 상담원이 고객에게 전화한다.
3. 상담 내용을 Word 파일에 기록한다.
4. 견적서, 면허증, 기타 서류는 Windows 폴더에 고객별로 저장한다.
5. 이후 과거 기록이나 파일을 찾을 때 Word와 탐색기를 다시 열어 확인한다.

현재 방식의 문제점은 다음과 같다.

* Word 파일을 여는 데 시간이 걸린다.
* 고객별 과거 상담 기록을 빠르게 찾기 어렵다.
* 견적 이미지와 상담 기록이 분리되어 있다.
* 면허증, 계약서, 기타 서류가 폴더 안에 흩어져 있다.
* 최근 상담 고객을 다시 찾기 어렵다.
* 파일명이나 폴더명에 의존해야 해서 검색성이 낮다.
* 비슷한 차량이나 조건의 과거 견적 기록을 찾아 참고하기 어렵다.

---

# 3. Goal

Consult Note의 목표는 다음과 같다.

* 고객별 상담 기록을 빠르게 작성하고 조회한다.
* 고객명, 전화번호, 차량명으로 빠르게 검색한다.
* 고객별 견적 이미지와 심사/계약 서류를 한 화면에서 확인한다.
* 비슷한 차량 또는 조건을 가진 과거 상담 기록과 견적을 빠르게 참고할 수 있게 한다.
* Word와 Windows 탐색기를 오가는 시간을 줄인다.
* 압축 해제 후 실행 파일만으로 사용할 수 있게 한다.

# 6. Platform Requirements

Consult Note는 Windows 전용 데스크톱 애플리케이션으로 개발한다.

권장 기술 스택:

* C#
* WPF
* SQLite
* Entity Framework Core
* Local File System

배포 요구사항:

* 압축 해제 후 실행 파일로 실행 가능해야 한다.
* Git 설치를 요구하지 않는다.
* Node.js 설치를 요구하지 않는다.
* npm 실행을 요구하지 않는다.
* .NET SDK 설치를 요구하지 않는다.
* 가능하면 self-contained publish 방식으로 배포한다.

예상 배포 구조:

```text
ConsultNote/
├── ConsultNote.exe
├── consultnote.db
├── storage/
│   └── customers/
├── backup/
└── logs/
```

# 7. Core Concepts

## 7.1 Customer

Customer는 상담 대상 고객을 의미한다.

Customer는 회사 DB의 고객 정보를 완전히 대체하지 않는다.
Consult Note 내부에서 상담 기록과 파일을 묶기 위한 단위이다.

주요 정보:

* 이름
* 전화번호
* 관심 차량
* 상태
* 상태 변경일
* 마지막 연락 시도일
* 메모
* 생성일
* 최근 상담일

고객 상태 예시:

* 상담중
* 부재
* 심사
* 계약완료
* 인도완료
* 장기부재
* 폐기

`부재` 상태가 오래 유지된 경우 MVP에서는 DB 상태를 자동 변경하지 않는다.
대신 화면에서 `부재 · 8일 경과`처럼 마지막 연락 시도일 또는 상태 변경일 기준 경과일을 표시할 수 있어야 한다.

---

## 7.2 Consultation Log

Consultation Log는 고객과의 상담 기록이다.

고객 1명은 여러 개의 상담 기록을 가질 수 있다.

예시:

```text
[2026-07-07 10:30]
K5 문의.
월 50만원 이하 희망.
배우자와 상의 후 재연락 예정.
```

상담 기록은 최신순 또는 오래된순으로 볼 수 있어야 한다.

---

## 7.3 Customer File

Customer File은 고객별로 보관하는 견적 이미지와 심사/계약 관련 서류이다.

견적은 여러 장 등록될 수 있으며, 파일 유형이 `견적`인 고객 파일로 관리한다.

파일 자체는 DB에 저장하지 않는다.
DB에는 파일 경로, 표시 파일명, 파일 유형 등 검색용 메타데이터만 저장한다.

필수 정보:

* 고객 ID
* 파일 경로
* 원본 파일명
* 저장 파일명
* 표시 파일명
* 파일 유형
* 저장일

선택 정보:

* 기타 파일 유형명
* 메모

파일 유형 기본값:

* 견적
* 면허증
* 사업자등록증
* 법인사업자등록증앞면
* 재무제표
* 부가세과세표준증명원
* 법인주주명부
* 법인등기부등본
* 법인 인감증명서
* 대표자 개인인감증명서
* 운전경력증명서
* 주민등록등본
* 소득증빙서류
* 계약서
* 기타

`기타` 선택 시 사용자가 기타 파일 유형명을 직접 입력할 수 있어야 한다.

프로그램은 견적 이미지나 서류 내용을 자동 분석하지 않는다.
차량명, 금액, 조건, 심사 관련 설명은 파일명 또는 상담 기록에 사용자가 남긴다.

필요 서류 예시:

* 캐피탈 개인: 면허증
* 캐피탈 개인사업자: 면허증, 사업자등록증
* 캐피탈 법인사업자: 법인사업자등록증앞면, 재무제표, 부가세과세표준증명원, 법인주주명부, 법인등기부등본, 대표님 운전면허증, 법인 인감증명서, 대표자 개인인감증명서
* 무심사/저신용 렌트사: 운전면허증, 운전경력증명서, 주민등록등본, 소득증빙서류

---

## 7.4 Vehicle Resource File

Vehicle Resource File은 특정 고객에게 묶이지 않는 공용 차량 자료이다.

예시:

* 미리 학습해둘 차량 견적서
* 여러 고객에게 홍보/판촉 목적으로 보낼 견적서
* 차량 설명용 특징 자료
* 공용 비교 참고 자료

공용 차량 자료는 고객 파일과 분리하여 `storage/vehicle-resources/` 폴더에 저장한다.

DB에는 파일 경로와 다음 메타데이터를 저장한다.

* 차량 제조사
* 차량명
* 유종
* 자료 유형
* 표시 파일명
* 메모
* 저장일

고객 화면의 같은 차량 자료 목록에서는 고객별 견적 파일과 공용 차량 자료를 함께 표시할 수 있어야 한다.

---

## 7.5 Vehicle

Vehicle은 상담 및 견적 입력 시 참고하는 차량명 목록이다.

차량명은 고객 정보와 견적 메타데이터에 문자열로 저장하되, 사용자가 매번 차량명을 새로 입력하지 않도록 로컬 DB에 차량명 목록을 보관한다.

주요 정보:

* 차량명
* 제조사 또는 브랜드
* 유종
* 메모
* 사용 여부
* 생성일
* 수정일

# 8. Functional Requirements

## 8.1 Customer Management

사용자는 고객 카드를 생성할 수 있어야 한다.

입력 항목:

* 이름
* 전화번호
* 관심 차량
* 메모

관심 차량은 차량명 목록에서 선택하거나 직접 입력할 수 있어야 한다.
직접 입력한 차량명은 차량명 목록에 추가할 수 있어야 한다.

사용자는 고객 정보를 수정할 수 있어야 한다.

사용자는 고객을 검색할 수 있어야 한다.

검색 기준:

* 이름
* 전화번호
* 차량명

고객 목록은 최근 상담일 기준으로 정렬할 수 있어야 한다.

고객 목록 정렬 기준:

* 등록순
* 최근 상담순
* 상태순

정렬 방향:

* 오름차순
* 내림차순

---

## 8.2 Consultation Log Management

사용자는 고객 상세 화면에서 상담 기록을 작성할 수 있어야 한다.

상담 기록 작성 시 현재 시간이 자동으로 저장되어야 한다.

사용자는 과거 상담 기록을 확인할 수 있어야 한다.

상담 기록은 고객 상세 화면에서 타임라인 형태로 표시되어야 한다.

MVP에서는 상담 기록 수정 및 삭제는 선택 기능으로 둔다.

---

## 8.3 Customer File Management

사용자는 고객별 파일을 추가할 수 있어야 한다.

지원 파일 형식:

* `.jpg`
* `.jpeg`
* `.png`
* `.pdf`

고객 파일 추가 시 사용자는 파일 유형을 드롭다운에서 선택할 수 있어야 한다.

파일 유형:

* 견적
* 면허증
* 사업자등록증
* 법인사업자등록증앞면
* 재무제표
* 부가세과세표준증명원
* 법인주주명부
* 법인등기부등본
* 법인 인감증명서
* 대표자 개인인감증명서
* 운전경력증명서
* 주민등록등본
* 소득증빙서류
* 계약서
* 기타

`기타` 선택 시 기타 파일 유형명을 직접 입력할 수 있어야 한다.

사용자는 표시용 파일명을 선택적으로 입력할 수 있다.
표시용 파일명이 비어 있으면 원본 파일명을 사용한다.
프로그램은 파일 내용을 자동 분석하지 않는다.

고객 상세 화면에서 고객 파일 목록을 확인할 수 있어야 한다.

고객 파일 목록에는 다음 정보가 표시된다.

* 저장일
* 파일 유형
* 표시 파일명

사용자는 파일 항목을 더블클릭하여 원본 파일을 열 수 있어야 한다.

필요 서류 예시:

* 캐피탈 개인: 면허증
* 캐피탈 개인사업자: 면허증, 사업자등록증
* 캐피탈 법인사업자: 법인사업자등록증앞면, 재무제표, 부가세과세표준증명원, 법인주주명부, 법인등기부등본, 대표님 운전면허증, 법인 인감증명서, 대표자 개인인감증명서
* 무심사/저신용 렌트사: 운전면허증, 운전경력증명서, 주민등록등본, 소득증빙서류

---

## 8.4 Search

검색은 프로그램의 핵심 기능이다.

사용자는 메인 화면에서 바로 검색할 수 있어야 한다.

검색 대상:

* 고객명
* 전화번호
* 관심 차량
* 상담 메모
* 고객 파일명

검색 결과는 빠르게 표시되어야 한다.

검색 결과에서 고객을 선택하면 고객 상세 화면으로 이동한다.

사용자는 비슷한 차량 또는 조건을 가진 다른 고객의 상담 기록과 견적을 검색 결과에서 빠르게 찾아 참고할 수 있어야 한다.

---

## 8.5 Vehicle Management

사용자는 차량 관리 화면에서 차량명 목록을 추가, 수정, 비활성화할 수 있어야 한다.

차량명 입력 화면은 차량명 목록을 기반으로 자동완성을 지원해야 한다.

비활성화된 차량은 고객 및 상담 조건 입력 목록에는 표시하지 않되, 차량 관리 화면에서는 다시 활성화할 수 있어야 한다.

---

## 8.6 Recent Customers

메인 화면에는 최근 상담 고객 목록이 표시되어야 한다.

최근 상담 고객은 `최근 상담일` 기준으로 정렬한다.

상담 기록이 추가되면 해당 고객의 최근 상담일이 자동으로 갱신되어야 한다.

# 9. Screen Requirements

## 9.1 Main Screen

메인 화면 구성:

* 검색창
* 최근 상담 고객 목록
* 고객 추가 버튼
* 고객 목록

고객 추가 시 이름은 필수로 입력받고, 연락처는 선택 입력으로 받는다.
차량 정보는 고객 추가 후 상담 조건 양식에서 입력한다.
아무 정보도 없는 빈 고객은 생성하지 않는다.

고객 목록 표시 정보:

* 이름
* 전화번호
* 고객 타입
* 관심 차량
* 유종
* 상태
* 최근 상담일
* 메모 일부

## 9.2 Customer Detail Screen

고객 상세 화면 구성:

* 상담 조건 양식
* 상담 기록 목록

상담 조건 양식과 상담 기록 쓰레드는 중앙 상세 영역에서 좌우로 배치한다.

주요 동작:

* 상담 조건 양식 작성 및 수정
* 상담 기록 추가
* 고객 파일 추가
* 파일 열기
* 저장 버튼을 통한 고객 기본 정보, 상담 조건 양식, 상태, 상담 기록 저장

상담 조건 양식 항목:

* 렌트/리스
* 차량스팩: 제조사, 차량명, 유종, 상세 옵션
* 계약기간: 기본값 60개월
* 주행거리: 기본값 2만km 또는 직접입력(0.0~5.0만km, 소수점 1자리)
* 인도지역: 미정, 서울/경기/인천, 부산/울산/경남, 대구/경북, 광주/전남, 전북, 대전/세종/충남, 충북, 강원, 제주
* 초기비용
* 명의형태
* 사업업력
* 실운전자
* 운전면허
* 보험연령
* 신용상태
* 특이사항

명의자 정보는 개인 또는 개인사업자 명의형태에서만 표시한다.
상담 기록 영역의 저장 버튼은 상담 조건 양식, 상태, 상담 기록을 함께 저장한다.

상담내용은 기본적으로 상담 기록 쓰레드로 관리한다.
쓰레드 상단에는 최신 상담내역 입력칸을 두고, 회사 사이트 첫통화 양식 복사 시 해당 최신 상담내역을 포함한다.

# 11. File Storage

파일은 DB에 직접 저장하지 않는다.

파일은 로컬 파일 시스템에 저장한다.

기본 저장 구조:

```text
storage/
└── customers/
    └── {customerId}/
        └── files/
            ├── 20260707_103000_a1b2c3d4.png
            ├── 20260707_104500_e5f6a7b8.jpg
            └── 20260707_110000_c9d0e1f2.pdf
```

DB에는 실제 파일 경로와 원본 파일명을 저장한다.

파일명 충돌을 방지하기 위해 저장 파일명은 시스템에서 자동 생성한다.

예시:

```text
yyyyMMdd_HHmmss_uniqueid.ext
```

# 12. Non-Functional Requirements

## 12.1 Performance

* 프로그램 실행은 가능한 한 빠르게 완료되어야 한다.
* 고객 검색 결과는 1초 이내에 표시되는 것을 목표로 한다.
* 고객 상세 화면은 1초 이내에 열리는 것을 목표로 한다.
* 상담 기록 저장은 즉시 완료되어야 한다.
* 파일 열기는 더블클릭 한 번으로 가능해야 한다.

## 12.2 Usability

* Word보다 빠르게 상담 기록을 열고 작성할 수 있어야 한다.
* 사용자는 고객을 찾기 위해 Windows 탐색기를 직접 열 필요가 없어야 한다.
* 자주 쓰는 기능은 메인 화면 또는 고객 상세 화면에서 바로 접근 가능해야 한다.
* 입력 항목은 최소화한다.
* 견적 등록 시 차량명, 월 렌트료, 상태, 메모 입력은 요구하지 않는다.

## 12.3 Reliability

* DB 저장 실패 시 사용자에게 오류를 표시해야 한다.
* 파일 복사 실패 시 사용자에게 오류를 표시해야 한다.
* 파일 경로가 깨졌을 경우 파일을 찾을 수 없다는 메시지를 표시해야 한다.

## 12.4 Backup

MVP에서는 자동 백업은 필수 기능이 아니다.

다만 데이터 구조는 수동 백업이 쉬워야 한다.

백업 대상:

```text
consultnote.db
storage/
```

사용자는 위 파일과 폴더를 복사하는 것만으로 데이터를 백업할 수 있어야 한다.

# 15. Development Principles

개발 시 다음 원칙을 따른다.

* 기능을 작게 나누어 구현한다.
* MVP 범위를 벗어나는 기능을 먼저 만들지 않는다.
* 로그인, 권한, 서버 기능을 임의로 추가하지 않는다.
* 파일은 DB에 저장하지 않는다.
* DB에는 파일 경로와 메타데이터만 저장한다.
* 사용자가 입력해야 하는 항목은 최소화한다.
* Word와 탐색기를 대체하는 속도와 편의성을 최우선으로 한다.
* 일반 사용자가 개발 도구 없이 실행할 수 있어야 한다.

---

# 16. Success Criteria

Consult Note의 1차 성공 기준은 다음과 같다.

* 사용자가 Word 파일을 열지 않고 상담 기록을 남길 수 있다.
* 고객 이름이나 전화번호로 1초 이내에 과거 상담을 찾을 수 있다.
* 차량명이나 견적 조건으로 과거 견적 기록을 빠르게 찾아 참고할 수 있다.
* 고객별 견적 이미지를 목록으로 볼 수 있다.
* 견적 이미지를 더블클릭으로 바로 열 수 있다.
* 면허증, 계약서 등 관련 파일을 고객 화면에서 바로 열 수 있다.
* 프로그램을 압축 해제 후 실행 파일만으로 사용할 수 있다.
