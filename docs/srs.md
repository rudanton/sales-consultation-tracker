# Consult Note

Version: 0.1
Target Platform: Windows
Application Type: Single-user Desktop App

---

# 1. Overview

Consult Note는 장기렌트 상담 업무를 위한 개인용 상담 기록 및 파일 관리 프로그램이다.

본 프로그램은 회사 CRM/DB를 대체하지 않는다.
회사의 고객 문의 DB는 기존처럼 사용하며, Consult Note는 상담원이 개인 PC에서 Word 문서와 고객별 폴더로 관리하던 상담 기록, 견적 이미지, 면허증, 계약 관련 파일을 더 빠르게 검색하고 열람하기 위한 보조 도구이다.

---

# 2. Problem

현재 업무 방식은 다음과 같다.

1. 회사 페이지에 고객 문의 DB가 들어온다.
2. 상담원이 고객에게 전화한다.
3. 상담 내용을 Word 파일에 기록한다.
4. 견적서, 면허증, 기타 서류는 Windows 폴더에 고객별로 저장한다.
5. 이후 과거 기록이나 파일을 찾을 때 Word와 탐색기를 다시 열어 확인한다.

현재 방식의 문제점은 다음과 같다.

* Word 파일을 여는 데 시간이 걸린다.
* 고객별 과거 상담 기록을 빠르게 찾기 어렵다.
* 견적 이미지와 상담 기록이 분리되어 있다.
* 면허증, 계약서, 기타 서류가 폴더 안에 흩어져 있다.
* 최근 상담 고객을 다시 찾기 어렵다.
* 파일명이나 폴더명에 의존해야 해서 검색성이 낮다.
* 비슷한 차량이나 조건의 과거 견적 기록을 찾아 참고하기 어렵다.

---

# 3. Goal

Consult Note의 목표는 다음과 같다.

* 고객별 상담 기록을 빠르게 작성하고 조회한다.
* 고객명, 전화번호, 차량명으로 빠르게 검색한다.
* 고객별 견적 이미지 목록을 한 화면에서 확인한다.
* 고객별 첨부파일을 한 화면에서 확인한다.
* 비슷한 차량 또는 조건을 가진 과거 상담 기록과 견적을 빠르게 참고할 수 있게 한다.
* Word와 Windows 탐색기를 오가는 시간을 줄인다.
* 압축 해제 후 실행 파일만으로 사용할 수 있게 한다.

# 6. Platform Requirements

Consult Note는 Windows 전용 데스크톱 애플리케이션으로 개발한다.

권장 기술 스택:

* C#
* WPF
* SQLite
* Entity Framework Core
* Local File System

배포 요구사항:

* 압축 해제 후 실행 파일로 실행 가능해야 한다.
* Git 설치를 요구하지 않는다.
* Node.js 설치를 요구하지 않는다.
* npm 실행을 요구하지 않는다.
* .NET SDK 설치를 요구하지 않는다.
* 가능하면 self-contained publish 방식으로 배포한다.

예상 배포 구조:

```text
ConsultNote/
├── ConsultNote.exe
├── consultnote.db
├── storage/
│   └── customers/
├── backup/
└── logs/
```

# 7. Core Concepts

## 7.1 Customer

Customer는 상담 대상 고객을 의미한다.

Customer는 회사 DB의 고객 정보를 완전히 대체하지 않는다.
Consult Note 내부에서 상담 기록과 파일을 묶기 위한 단위이다.

주요 정보:

* 이름
* 전화번호
* 관심 차량
* 회사 DB 번호 또는 링크
* 상태
* 상태 변경일
* 마지막 연락 시도일
* 메모
* 생성일
* 최근 상담일

고객 상태 예시:

* 상담중
* 부재
* 심사
* 계약완료
* 인도완료
* 장기부재
* 폐기

`부재` 상태가 오래 유지된 경우 MVP에서는 DB 상태를 자동 변경하지 않는다.
대신 화면에서 `부재 · 8일 경과`처럼 마지막 연락 시도일 또는 상태 변경일 기준 경과일을 표시할 수 있어야 한다.

---

## 7.2 Consultation Log

Consultation Log는 고객과의 상담 기록이다.

고객 1명은 여러 개의 상담 기록을 가질 수 있다.

예시:

```text
[2026-07-07 10:30]
K5 문의.
월 50만원 이하 희망.
배우자와 상의 후 재연락 예정.
```

상담 기록은 최신순 또는 오래된순으로 볼 수 있어야 한다.

---

## 7.3 Estimate

Estimate는 고객에게 안내한 견적 이미지이다.

견적서는 주로 `.jpg`, `.png` 이미지 파일로 저장된다.

파일 자체는 DB에 저장하지 않는다.
DB에는 파일 경로와 검색용 메타데이터만 저장한다.

필수 정보:

* 고객 ID
* 이미지 파일 경로
* 원본 파일명
* 저장일

선택 정보:

* 차량명
* 월 렌트료
* 상태
* 메모

견적 상태 예시:

* 초안
* 안내완료
* 보류
* 최종

---

## 7.4 Attachment

Attachment는 견적 외의 고객 관련 파일이다.

예시:

* 면허증
* 계약서
* 신분증
* 사업자등록증
* 기타 서류

파일 자체는 로컬 폴더에 저장하고, DB에는 파일 경로만 저장한다.

# 8. Functional Requirements

## 8.1 Customer Management

사용자는 고객 카드를 생성할 수 있어야 한다.

입력 항목:

* 이름
* 전화번호
* 관심 차량
* 회사 DB 번호 또는 링크
* 메모

사용자는 고객 정보를 수정할 수 있어야 한다.

사용자는 고객을 검색할 수 있어야 한다.

검색 기준:

* 이름
* 전화번호
* 차량명

고객 목록은 최근 상담일 기준으로 정렬할 수 있어야 한다.

고객 목록 정렬 기준:

* 등록순
* 최근 상담순
* 상태순

정렬 방향:

* 오름차순
* 내림차순

---

## 8.2 Consultation Log Management

사용자는 고객 상세 화면에서 상담 기록을 작성할 수 있어야 한다.

상담 기록 작성 시 현재 시간이 자동으로 저장되어야 한다.

사용자는 과거 상담 기록을 확인할 수 있어야 한다.

상담 기록은 고객 상세 화면에서 타임라인 형태로 표시되어야 한다.

MVP에서는 상담 기록 수정 및 삭제는 선택 기능으로 둔다.

---

## 8.3 Estimate Management

사용자는 고객에게 견적 이미지를 추가할 수 있어야 한다.

지원 파일 형식:

* `.jpg`
* `.jpeg`
* `.png`

견적 추가 시 사용자는 다음 정보를 선택적으로 입력할 수 있다.

* 차량명
* 월 렌트료
* 상태
* 메모

고객 상세 화면에서 견적 목록을 확인할 수 있어야 한다.

견적 목록에는 다음 정보가 표시된다.

* 저장일
* 차량명
* 월 렌트료
* 상태
* 메모 일부

사용자는 견적 항목을 더블클릭하여 원본 이미지를 열 수 있어야 한다.

---

## 8.4 Attachment Management

사용자는 고객 상세 화면에서 첨부파일을 추가할 수 있어야 한다.

지원 파일 형식은 제한하지 않는다.

첨부파일 예시:

* 면허증 이미지
* 계약서 PDF
* 기타 서류 이미지
* 엑셀 파일
* 워드 파일

사용자는 첨부파일 목록을 확인할 수 있어야 한다.

사용자는 첨부파일을 더블클릭하여 기본 프로그램으로 열 수 있어야 한다.

---

## 8.5 Search

검색은 프로그램의 핵심 기능이다.

사용자는 메인 화면에서 바로 검색할 수 있어야 한다.

검색 대상:

* 고객명
* 전화번호
* 관심 차량
* 상담 메모
* 견적 차량명

검색 결과는 빠르게 표시되어야 한다.

검색 결과에서 고객을 선택하면 고객 상세 화면으로 이동한다.

사용자는 비슷한 차량 또는 조건을 가진 다른 고객의 상담 기록과 견적을 검색 결과에서 빠르게 찾아 참고할 수 있어야 한다.

---

## 8.6 Recent Customers

메인 화면에는 최근 상담 고객 목록이 표시되어야 한다.

최근 상담 고객은 `최근 상담일` 기준으로 정렬한다.

상담 기록이 추가되면 해당 고객의 최근 상담일이 자동으로 갱신되어야 한다.

# 11. File Storage

파일은 DB에 직접 저장하지 않는다.

파일은 로컬 파일 시스템에 저장한다.

기본 저장 구조:

```text
storage/
└── customers/
    └── {customerId}/
        ├── estimates/
        │   ├── estimate_20260707_103000.png
        │   └── estimate_20260708_141500.jpg
        └── attachments/
            ├── license.jpg
            ├── contract.pdf
            └── etc.png
```

DB에는 실제 파일 경로와 원본 파일명을 저장한다.

파일명 충돌을 방지하기 위해 저장 파일명은 시스템에서 자동 생성한다.

예시:

```text
estimate_yyyyMMdd_HHmmss_originalname.png
```

# 12. Non-Functional Requirements

## 12.1 Performance

* 프로그램 실행은 가능한 한 빠르게 완료되어야 한다.
* 고객 검색 결과는 1초 이내에 표시되는 것을 목표로 한다.
* 고객 상세 화면은 1초 이내에 열리는 것을 목표로 한다.
* 상담 기록 저장은 즉시 완료되어야 한다.
* 파일 열기는 더블클릭 한 번으로 가능해야 한다.

## 12.2 Usability

* Word보다 빠르게 상담 기록을 열고 작성할 수 있어야 한다.
* 사용자는 고객을 찾기 위해 Windows 탐색기를 직접 열 필요가 없어야 한다.
* 자주 쓰는 기능은 메인 화면 또는 고객 상세 화면에서 바로 접근 가능해야 한다.
* 입력 항목은 최소화한다.
* 견적 등록 시 차량명이나 월 렌트료 입력은 선택 사항이어야 한다.

## 12.3 Reliability

* DB 저장 실패 시 사용자에게 오류를 표시해야 한다.
* 파일 복사 실패 시 사용자에게 오류를 표시해야 한다.
* 파일 경로가 깨졌을 경우 파일을 찾을 수 없다는 메시지를 표시해야 한다.

## 12.4 Backup

MVP에서는 자동 백업은 필수 기능이 아니다.

다만 데이터 구조는 수동 백업이 쉬워야 한다.

백업 대상:

```text
consultnote.db
storage/
```

사용자는 위 파일과 폴더를 복사하는 것만으로 데이터를 백업할 수 있어야 한다.

# 15. Development Principles

개발 시 다음 원칙을 따른다.

* 기능을 작게 나누어 구현한다.
* MVP 범위를 벗어나는 기능을 먼저 만들지 않는다.
* 로그인, 권한, 서버 기능을 임의로 추가하지 않는다.
* 파일은 DB에 저장하지 않는다.
* DB에는 파일 경로와 메타데이터만 저장한다.
* 사용자가 입력해야 하는 항목은 최소화한다.
* Word와 탐색기를 대체하는 속도와 편의성을 최우선으로 한다.
* 일반 사용자가 개발 도구 없이 실행할 수 있어야 한다.

# 16. Success Criteria

Consult Note의 1차 성공 기준은 다음과 같다.

* 사용자가 Word 파일을 열지 않고 상담 기록을 남길 수 있다.
* 고객 이름이나 전화번호로 1초 이내에 과거 상담을 찾을 수 있다.
* 차량명이나 견적 조건으로 과거 견적 기록을 빠르게 찾아 참고할 수 있다.
* 고객별 견적 이미지를 목록으로 볼 수 있다.
* 견적 이미지를 더블클릭으로 바로 열 수 있다.
* 면허증, 계약서 등 관련 파일을 고객 화면에서 바로 열 수 있다.
* 프로그램을 압축 해제 후 실행 파일만으로 사용할 수 있다.
